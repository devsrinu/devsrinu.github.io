The math behind symmetric keys generation in SSL handshake

Learn how symmetric keys are generated in SSL handshake

Background

The high level idea of SSL (the technology to keep your internet connection secure) is to encrypt the data that is exchanged between the server and the client.

Naturally the next question is: what would be used to encrypt the data?

There are two possibilities:

  1. Use assymetric key encryption
  2. Use a symmetric key

Assymetric key encryption has the advantage of using public (which is of course public) key of your recipient to encrypt the data. Recipient then uses his secret private key to decrypt the data.

Whats the downside?

The risk is that if an attacker is recording the encryted messages. In the future if the attacker manages to get the private key of the recipient, he/she would be able to decrypt the previously recorded encrypted messages.

That’s where ephemeral symmetric keys are useful.

The idea is to generate short lived symmetric key and use that to encrypt, decrypt the messages.

This achieves what is called as Forward Secrecy that protects past sessions against future compromises of secret keys.

This post is about giving a high level overview of the math (using an example) involved in generating a symmetric key.

The key aspect of this particular approach is that it doesn’t require any prior knowledge of each other.

The Math

The math involves the following one-way function: $$Y^X(mod P)$$ The involved parties agree upon values for $Y$ and $P$. In reality the values are fixed and very large prime numbers.

The basic premise is that its very hard to compute (or reverse engineer) the value of $X$ given the result of $Y^X (mod P)$ is provided (even though $Y$ and $P$ are not secret)

This is called Discrete Log problem.

You can watch more about this here -> https://www.youtube.com/watch?v=BRMj5jE6Z-U

The beauty of this is that the $mod$ operation makes it easy to generate the results but makes it hard to reverse engineer or find the value of $X$.

For the sake of understanding we will have the following setup:

  • We will use $P=11$ and $Y=7$ (both need to be primes)
  • Let’s call the involved parties Alice and Bob

Alice does the following:

  • Chooses a number say $3$ and keeps it secret (lets call it $A$)
  • Alice runs the number through the one-way function and works out the result:
    • $7^3 (mod 11)$ = $343 (mod 11)$ = $2$
    • Alice calls this $α$
  • Alice sends the result ($α=2$) to Bob

Bob does something similar during this time:

  • Chooses a number say $6$ and keeps it secret (lets call it $B$)
  • Bob also runs the number through the same one-way function and works out the result:
    • $7^6 (mod 11)$ = $117,649 (mod 11)$ = $4$
    • Bob calls this $β$
  • Bob sends the result ($β=4$) to Bob

The final step

Now both Alice and Bob have got the result of one-way function from the other party.

They are ready to compute the symmetric key.

This is what Alice does:

Alice works out the result of another one-way function that uses the secret ($A=3$) and the result from Bob ($β=4$) as follows:

  • Function used: $β^A(mod 11)$
  • Evaluation: $4^3 (mod 11)$ = $64 (mod 11)$ = $9$
  • Alice would use $9$ as the symmetric key to encrypt/decrypt messages to/from Bob

Meanwhile Bob does something similar:

He works out result of a similar one-way function with his secret ($B=6$) and the result from Alice ($α=2$):

  • Function used: $α^B (mod 11)$
  • Evaluation: $2^6 (mod 11)$ = $64 (mod 11)$ = $9$
  • Bob also ended up with the same key $9$ and uses that to encrypt/decrypt the messages to/from Alice

Further Reading

The above is formally called an ephemeral Diffie-Hellman handshake

This particular part of the SSL handshake is called Key Exchange Algorithm

If the client and server use TLS 1.3 then it enforces usage of this algorithm for key exchange.
